What is GOBBLE?

GOBBLE is a webcam-based arcade game built for the tio.ist co-working space in Kadıköy. Players eat falling pixel-art food items by opening their mouths in front of a webcam. There is no keyboard, no controller — your face is the input.

It runs entirely in the browser as a single HTML file with no build step, no dependencies served from npm, and no backend. Every sprite, sound, and UI element is generated at runtime in JavaScript.

Two modes exist:

  • Solo (/gobble/) — one player, score attack with a local leaderboard.
  • Duello (/gobble/duello/) — two players side by side, competing for 45 seconds.

Technology Stack

MediaPipe FaceLandmarker

Face detection runs on the MediaPipe FaceLandmarker task (WASM + WebGL GPU delegate). It outputs 478 landmarks per detected face. Key landmarks used:

  • Landmark 4 — nose tip. Used for face position and zone assignment. Stable across all mouth states.
  • Landmarks 13, 14 — upper/lower lip. Used to measure vertical mouth opening.
  • Landmarks 61, 291 — left/right mouth corner. Used to measure mouth width.
// Mouth-open detection: ratio of height to width > 0.28
const mH = Math.hypot((ul.x - ll.x) * cx, (ul.y - ll.y) * cy);
const mW = Math.hypot((lc.x - rc.x) * cx, (lc.y - rc.y) * cy);
m.open = mW > 0 && (mH / mW) > 0.28;

Web Audio API

All sound is synthesised at runtime using the Web Audio API — no audio files are loaded. An AudioContext is created lazily on the first user gesture to comply with browser autoplay policy.

  • SFX — short OscillatorNode bursts with gain envelopes and optional frequency bends.
  • Music — a 16-step minor-pentatonic chiptune loop scheduled ahead of the current time via tickMusic(), called every game frame.
  • Fever mode — BPM doubles from 128 to 200 when fever is active.
// Lazy init on first tap/click
document.addEventListener('click', initAudio, { once: true });
document.addEventListener('touchstart', initAudio, { once: true });

function initAudio() {
  if (audioCtx) return;
  audioCtx = new AudioContext();
  masterGain = audioCtx.createGain();
  masterGain.gain.value = 0.8;
  masterGain.connect(audioCtx.destination);
}

Canvas 2D

Everything visible during gameplay is drawn on a single <canvas> element that covers the full viewport. The live webcam feed is rendered in a mirrored <video> behind the canvas.

Sprites are SVG pixel-art definitions converted to Blob URLs and pre-loaded into an Image cache before the game starts. This means drawImage() calls are GPU-accelerated with no per-frame SVG parsing. 

// Pre-rasterise SVG sprites into Image cache
function cacheSprites() {
  return Promise.all([...GOOD_KEYS, 'poop'].map(k => new Promise(res => {
    const img = new Image();
    const blob = new Blob([mkSvg(52, PA[k])], { type: 'image/svg+xml' });
    img.onload = () => { spriteCache[k] = img; res(); };
    img.src = URL.createObjectURL(blob);
  })));
}

No build step

Each game mode is a self-contained HTML file. MediaPipe is loaded from a CDN (cdn.jsdelivr.net). Fonts come from Google Fonts. Zero npm packages, zero bundlers. Deploy = copy file to S3.

Why does duello.html exist?
gobble.tio.ist is served from an S3 bucket via CloudFront. When a visitor hits the extensionless path /duello, CloudFront looks for an exact key match. Without duello.html at the bucket root, the request would 404 (or fall through to the custom error page). duello.html is a mirror of duello/index.html — it is not trash. Both files serve the same game; the subdirectory version handles /duello/ (with trailing slash) while the root file handles /duello (without).

Zone assignment (Duello-specific)

The webcam is mirrored horizontally (transform: scaleX(-1)). A person standing on the physical right side of the camera appears on the left side of the video. Zone assignment uses the nose tip (landmark 4) for its position stability: 

// screenX = 1 − camera X (because video is mirrored)
const screenX = 1 - lm[4].x;

// screenX < 0.5  → face on left  → P1 (mouth)
// screenX >= 0.5 → face on right → P2 (mouth2)
const m = (!TEST_MODE && screenX >= 0.5) ? mouth2 : mouth;

Lip landmarks (13/14) were deliberately not used for zone assignment. When a mouth opens, lip X-coordinates shift, which previously caused P1 and P2 to swap on every frame — freezing item updates as each assignment overrode the other. The nose tip is rigid and does not move with the jaw.

Once waitState transitions to 'counting', it is permanently locked until the game starts. Detection glitches (MediaPipe occasionally dropping a face for one frame) cannot reset the countdown: 

if (waitState === 'counting') {
  // LOCKED — ignore face-loss events
} else {
  const bothSides = faceInLeft && faceInRight;
  if (bothSides) {
    stableMs += dt * 1000;
    if (stableMs >= STABLE_NEEDED) {
      waitState = 'counting'; autoCountdown = 5;
    }
  } else {
    stableMs = Math.max(0, stableMs - dt * 600);
  }
}

Why Items Get Stuck When a Player Opens Their Mouth

This was the primary bug during development. Items would visually freeze (stop falling) whenever a player opened their mouth, then resume when they closed it. The game loop was still running — only the item updates halted. Root cause was a cascade:

  1. Using lip landmarks for zone assignment. When a player opened their mouth, lip X-coordinates shifted enough to flip which zone their face was assigned to. This caused mouth and mouth2 to be overwritten in the wrong order, effectively clearing the active player's mouth state every other frame.
    Fix: use nose tip (landmark 4) for zone assignment instead.
  2. sfx() throwing when masterGain was null. Before audio was initialised by a user gesture, masterGain was null. sfx() tried to call masterGain.gain.setValueAtTime() and threw. A blanket try/catch around the game update loop swallowed the error silently, but the item had already been marked collected = true before collectItem() was called, so the scoring, particles and floating text never ran. Items disappeared from the spawn list but showed no visual feedback.
    Fix: guard sfx() against null early, remove the blanket catch. 
// Guard prevents throw when AudioContext isn't ready yet
function sfx(freq, type, dur, vol = 0.28, bend = null) {
  if (!audioCtx || !masterGain || audioCtx.state !== 'running') return;
  // rate-limit: no more than one SFX every 40 ms
  const now100 = Date.now();
  if (now100 - _lastSfxTime < 40) return;
  _lastSfxTime = now100;
  // ... create oscillator
}

Performance Notes

  • Sprite cache — SVG→Blob→Image pre-loaded once before start. drawImage() calls are GPU-accelerated, no per-frame SVG parse.
  • Detection throttle — face detection runs every 80 ms during the wait phase, 28 ms during gameplay. This prevents detection from blocking the render frame.
  • No DOM updates during gameplay — scores, particles, and floating text are all drawn on the canvas. Zero DOM mutations = zero layout recalculations per frame.
  • Particle/text pools — arrays are filtered in-place each frame (p.life > 0), avoiding allocations in the hot path.
  • Audio rate limit — sfx() enforces a 40 ms cooldown to prevent Web Audio graph overflow when many items are collected simultaneously.

Test Mode

Duello requires two people, which makes local development awkward. Add ?test to the URL to enable single-player testing: 

https://gobble.tio.ist/duello/?test

In test mode:

  • A single detected face unlocks both zones in the wait state (countdown starts immediately).
  • The single mouth is mirrored to mouth2 each frame, so both P1 and P2 zones are active.
  • Both halves of the screen are playable by one person.
const TEST_MODE = new URLSearchParams(location.search).has('test');

// In wait state: 1 face counts as both zones
const bothSides = TEST_MODE
  ? (faceInLeft || faceInRight)
  : (faceInLeft && faceInRight);

// In game loop: mirror single face to P2
if (TEST_MODE && mouth.active && !mouth2.active) {
  mouth2.x = mouth.x; mouth2.y = mouth.y;
  mouth2.open = mouth.open; mouth2.active = true;
}

Main Game Loop

function mainLoop(ts) {
  requestAnimationFrame(mainLoop);

  // 1. Run face detection (throttled)
  runDetection(ts);

  if (!gameActive) {
    // 2a. Wait phase: render detection UI + handle auto-countdown
    handleWait(ts);
    renderWait();
  } else {
    // 2b. Game phase: music → physics → render
    const dt = Math.min((ts - lastTime) / 1000, 0.05);
    lastTime = ts;
    tickMusic();
    updateGame(dt);  // items, scoring, fever
    renderGame();   // canvas draw
  }
}

The frame delta is clamped to 50 ms (20 fps minimum) so a tab backgrounded briefly does not cause items to teleport on resume.
